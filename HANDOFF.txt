Users provide a product URL (not a search term).
Bot detects platform from domain, blocks known anti-scrape platforms (Nike, Adidas,
ASOS, Amazon, Zara, H&M) with a helpful message.

  - Scrapes JSON-LD structured data → Shopify /products/{handle}.js → meta tags → Next.js __NEXT_DATA__

  - Stores full variant baseline in Supabase settings
  - Cron job re-scrapes and alerts on any per-variant price drop via Discord DM
  - Legacy text-based watchlist items still work (Depop/Vinted search)

── CREDITS SYSTEM (April 17 session) ──

Credits replace the per-command rate limits for paid AI generation.
80 credits = one PhotoRoom photo enhancement.

Database:
  profiles.credits                INTEGER NOT NULL DEFAULT 0
  credit_codes                    redeemable codes (admin-managed)
  code_redemptions                UNIQUE (code, user_id) — prevents double-redeem

Helpers (bot/index.js):
  getCredits(userId)              SELECT profiles.credits WHERE id = userId (UUID)
  setCredits(userId, amount)      PATCH profiles SET credits = max(0, round(amount))
  addCredits(userId, amount)      get + set, returns new balance
  deductCredits(userId, amount)   throws { code:'INSUFFICIENT_CREDITS', balance } if low

Endpoints:
  GET  /api/credits/balance               → { credits: N }
  POST /api/credits/create-order          → creates PayPal order, custom_id = "uid|pkg"
  POST /api/credits/capture-order         → captures payment, credits user, returns newBalance
  POST /api/credits/redeem                → redeems a credit code, records redemption
  POST /api/photo/photoroom               → deducts 80 cr, calls PhotoRoom, refunds on fail
  POST /api/admin/credits/adjust          → owner-only; accepts Discord ID OR Supabase UUID,
                                            resolves Discord ID via profiles.discord_id
                                            lookup, then set/add/subtract
  GET  /api/admin/codes                   → owner-only, lists all credit codes
  POST /api/admin/codes                   → owner-only, creates a credit code
  DELETE /api/admin/codes/:code           → owner-only, deletes a code

── VINTED PROXY / DATADOME WORK (April 16 session — legacy fallback path) ──

Significant work done to get Vinted listing past DataDome. Key fixes:

1. Global Node 18 fetch silently ignores the `dispatcher` option — proxy was never
   actually applied. Fixed by importing undici's own fetch (undFetch) and routing all
   Vinted calls through vFetch() wrapper that uses undFetch when PROXY_AGENT is active.

2. One global PROXY_AGENT created at startup (not a new instance per request).
   Rotating proxies caused IP changes mid-session; DataDome detects this and blocks.
   Single shared agent = consistent exit IP for entire process lifetime.

3. Chrome TLS cipher config added to ProxyAgent to mimic real browser TLS fingerprint.

4. getVintedBase() and getVintedSession() split into two separate functions to fix crash.

5. Manual token validation: /api/vinted/validate-token endpoint added.
   Dashboard shows manual token input UI; Depop shows "coming soon" message instead.

6. Geo-redirect fix: Vinted may redirect to .fr/.de etc. via geo; session bootstrap
   now follows redirect and locks the correct base domain.

7. Token expiry detection narrowed to exact Vinted message_code values:
   "unauthenticated" or "invalid_auth_token" only.
   (Previously matched too broadly — locale mismatches and scope errors were
    wrongly flagged as expired tokens, confusing users with reconnect prompts.)

8. ByteString crash fixed: header values now sanitised before passing to undici.

9. Detailed DataDome logging added to distinguish:
   "proxy active but IP flagged" vs "no proxy set at all"
   Bot logs proxy status on startup.

Status: The legacy vFetch path remains as a fallback but is NO LONGER the
primary route. All Vinted traffic now goes through the Playwright browser
flow. DataDome solved-cookie is persisted inside the browser context for
the process lifetime.
